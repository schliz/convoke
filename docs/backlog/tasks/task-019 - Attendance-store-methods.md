## Description

<!-- SECTION:DESCRIPTION:BEGIN -->
Implement store-layer methods for attendance records in internal/store/. Follow the existing DBTX pattern.

Required store methods:
- CreateAttendance(ctx, db, entryID, userID, status) — create an attendance record
- UpdateAttendanceStatus(ctx, db, entryID, userID, status) — change attendance status
- DeleteAttendance(ctx, db, entryID, userID) — remove attendance record
- GetAttendance(ctx, db, entryID, userID) — get a user's attendance for an entry
- ListAttendeesByEntry(ctx, db, entryID) — list all attendance records for an entry (with user info via JOIN)
- CountAttendeesByStatus(ctx, db, entryID, status) — count attendees with a given status (accepted, declined, pending)
- CountAcceptedAttendees(ctx, db, entryID) — shorthand for counting accepted (used for staffing checks)
- IsAttendee(ctx, db, entryID, userID) — check if user has any attendance record
- BulkCreatePendingAttendance(ctx, db, entryID, userIDs) — create pending records for a list of users (for meeting audience)
- ListEntriesWithUserAttendance(ctx, db, userID, startDate, endDate) — list entries with the user's attendance status attached (for personal dashboard)

The UNIQUE(entry_id, user_id) constraint on the attendance table means CreateAttendance should handle the conflict case (either error or upsert, depending on the caller's needs).

These methods are performance-critical since they're called on every entry card render. CountAcceptedAttendees in particular should be efficient.

# TASK-019: Attendance Store Methods — Implementation Plan

## 1. Context Summary

### Project Architecture

Convoke is a Go web application using server-side rendering (Go templates + HTMX) backed by PostgreSQL via pgx/v5. The database query layer is generated by **sqlc v1.30.0**:

- SQL query definitions live in `internal/store/queries/*.sql` (one file per entity).
- sqlc generates Go code into `internal/db/` (package `db`), producing a `DBTX` interface, a `Queries` struct, model types, and typed query functions.
- The `internal/store/` package wraps `db.Queries` via a `Store` struct that holds a `*pgxpool.Pool` and provides `Queries()`, `Pool()`, and `WithTx()` methods.
- Handlers access the store via `h.Store.Queries().SomeMethod(ctx, params)`.
- For transactional operations, handlers use `h.Store.WithTx(ctx, func(tx pgx.Tx, q *db.Queries) error { ... })`.

### Current Store Pattern (main repo)

The main repo has evolved away from the worktree's DBTX-as-first-arg pattern. In the current main repo:

- `internal/store/store.go` defines `Store` with `pool` and `queries *db.Queries`.
- All query functions are **methods on `*db.Queries`**, not package-level functions.
- `db.Queries` itself accepts a `DBTX` interface, and `db.New(dbtx)` creates a scoped instance.
- `Store.WithTx()` provides `func(tx pgx.Tx, q *db.Queries) error` — the callback receives a transaction-scoped `*db.Queries`.

**Therefore, this task should add new sqlc query definitions** that sqlc will generate into methods on `*db.Queries`. No hand-written store wrapper file is needed — the generated code IS the store layer. Higher-level composition (e.g., `BulkCreatePendingAttendance` which loops over users) will be a thin method on `*Store` or a standalone function in `internal/store/` that calls the generated queries.

### Attendance Table Schema

From `migrations/00007_attendance.sql`:

```sql
CREATE TABLE attendances (
    id           BIGSERIAL    PRIMARY KEY,
    entry_id     BIGINT       NOT NULL REFERENCES entries(id) ON DELETE CASCADE,
    user_id      BIGINT       NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    status       TEXT         NOT NULL DEFAULT 'pending'
        CHECK (status IN ('pending', 'accepted', 'declined', 'needs_substitute', 'replaced')),
    confirmed    BOOLEAN      NOT NULL DEFAULT false,
    note         TEXT,
    responded_at TIMESTAMPTZ,
    created_at   TIMESTAMPTZ  NOT NULL DEFAULT now(),
    updated_at   TIMESTAMPTZ  NOT NULL DEFAULT now()
);

CREATE UNIQUE INDEX idx_attendances_entry_user ON attendances (entry_id, user_id);
CREATE INDEX idx_attendances_entry_id ON attendances (entry_id);
CREATE INDEX idx_attendances_user_id ON attendances (user_id);
CREATE INDEX idx_attendances_user_status ON attendances (user_id, status);
```

Key facts:
- `UNIQUE(entry_id, user_id)` enforced via unique index — a user can have at most one attendance per entry.
- Status is a CHECK constraint (not a Postgres enum type), stored as `TEXT`. Valid values: `pending`, `accepted`, `declined`, `needs_substitute`, `replaced`.
- `responded_at` records when the user last changed from pending status.
- `confirmed` is an admin-set flag for post-hoc attendance confirmation (deferred in v1 per requirements).
- CASCADE deletes: deleting an entry or user removes all associated attendance records.

### Design Document Insights

From the requirements document (section 3.6 — Attendance):

- **Shifts**: attendance records are created on demand (user signs up). No pre-population.
- **Meetings**: attendance records are pre-created as `pending` for every user in the audience. This is the primary use case for `BulkCreatePendingAttendance`.
- Staffing/response status is **computed, not stored** — derived from counting attendance records.
- The `confirmed` field is for post-hoc admin verification (deferred for v1).
- Performance is critical: attendance counts are displayed on every entry card in calendar views.
